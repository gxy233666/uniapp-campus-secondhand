## 二、主要问题与风险

### P1：交付文档曾经明显滞后，可能影响评分

问题说明：

老师要求提交 README、API 文档、Prompt 记录、AI Code Review 报告等材料。审查时发现早期 `README.md` 和 `docs/api.md` 仍偏早期版本，缺少注册登录、院校筛选、交易意向、图片上传、已售出等新功能说明；`docs/prompt_log.md` 和 `docs/code_review.md` 当时仍是模板。

影响：

如果不补齐文档，即使功能完成，也可能在“AI 工具与过程文档”“工程规范与代码质量”部分丢分。

处理建议：

- 更新 README，补齐项目简介、技术栈、功能清单、运行方式、uniCloud 配置、部署地址、演示账号、注意事项。
- 更新 API 文档，按 `user/product/favorite/intent/school/upload` 分类说明接口。
- 补充 Prompt 日志，记录关键提示词、AI 输出摘要和对应功能。
- 将本 Code Review 报告保存到 `docs/code_review.md`，并保留截图。

当前状态：已处理。

### P1：后端较多接口信任前端传入的用户身份

问题说明：

当前登录状态主要保存在前端本地缓存中。发布、编辑、下架、删除、标记已售出、交易意向状态更新等操作，后端主要依赖前端传来的 `userId` 或 `seller_id` 判断权限。

涉及位置：

- `pages/publish/publish.vue`
- `common/api.js`
- `uniCloud-aliyun/cloudfunctions/product/index.obj.js`
- `uniCloud-aliyun/cloudfunctions/intent/index.obj.js`

影响：

课程演示场景下可以接受，但如果项目公开给大量用户使用，理论上用户可以通过修改前端请求参数伪造身份，操作不属于自己的商品或交易意向。

处理建议：

- 如果作为课程演示项目，可以在 README 中说明当前身份机制适合演示环境。
- 如果后续要正式上线，应接入 `uni-id` 或其他服务端登录态校验机制。
- 后端应从服务端登录态中获取当前用户 ID，而不是完全信任前端传入的用户 ID。

当前状态：已在 README 注意事项中说明，后续可作为增强功能继续完善。

### P1：数据库 schema 权限偏开放

问题说明：

部分数据库集合 schema 允许较宽的 `read/create/update` 权限。当前项目主要通过云对象访问数据库，但如果存在直接数据库访问路径，权限过宽会带来安全风险。

涉及集合：

- `products`
- `users`
- `favorites`
- `intents`
- `schools`

影响：

正式环境下可能出现非预期的数据读写风险。

处理建议：

- 课程演示阶段可以保留当前配置，方便调试和演示。
- 如果正式部署，应收紧 schema 权限。
- 关键写操作统一走云对象，由云对象做权限校验。

当前状态：暂不修改，已作为注意事项记录。

### P2：云存储图片在 H5 发行后需要再次验证

问题说明：

发布商品时使用 `uniCloud.uploadFile()` 上传图片，并将返回的 `fileID` 保存到商品数据的 `image_url` 字段。页面中直接使用 `image_url` 展示图片。

涉及位置：

- `pages/publish/publish.vue`
- `pages/index/index.vue`
- `pages/product-detail/product-detail.vue`
- `pages/mine/mine.vue`

影响：

在本地调试中可能正常，但 H5 发行后仍需要确认云存储图片是否能被正确访问。如果 H5 端无法直接显示 `fileID`，商品图片可能显示为空。

处理建议：

- H5 发行后测试“发布商品 + 上传图片 + 首页显示 + 详情页显示”。
- 如果不能直接显示，应使用 `uniCloud.getTempFileURL()` 将 `fileID` 转换为临时访问地址。

当前状态：功能已实现，待 H5 部署后验证。

### P2：缺少自动化测试或固定验收清单

问题说明：

项目目前主要依靠手动运行和页面测试，没有自动化测试脚本。

影响：

课程项目可以接受，但最后演示前容易遗漏关键流程，比如注册、发布、收藏、交易意向、卖家同意、标记已售出等。

处理建议：

准备一份手动验收清单，按顺序检查：

1. 注册买家账号。
2. 注册卖家账号。
3. 卖家发布商品并上传图片。
4. 首页可以看到商品。
5. 买家收藏商品。
6. 买家提交交易意向。
7. 卖家查看收到意向并同意。
8. 买家查看联系方式。
9. 卖家标记商品已售出。
10. 首页不再把已售出商品作为可购买商品展示。

当前状态：建议在录制演示视频前完成一次手动验收。

## 四、优化建议

### 1. 提交前必须完成

- 补充 H5 线上访问地址。
- 录制项目演示视频。
- 编写个人总结，字数不少于 500 字。
- 准备数据库、API、AI Code Review 截图包。
- 在最终提交前完整跑一遍核心流程。

### 2. 可以作为加分优化

- 进一步精修首页和我的页面 UI。
- 在详情页增加更明显的交易状态提示。
- 对云数据库 `resource exhausted` 这类错误提供更友好的用户提示。
- 对图片上传失败、图片过大等情况增加提示。
- 将身份认证升级为 `uni-id`。
- 收紧数据库 schema 权限。

### 3. 不建议最后阶段大改

- 不建议临近提交前接入真实支付。
- 不建议临近提交前重构整个 UI 框架。
- 不建议更换后端方案。
- 不建议引入过多新依赖。

原因是当前项目已经形成完整闭环，最后阶段应优先保证稳定性和材料完整性。

## 五、最终判断

本项目已经达到课程项目的基本完整度要求。功能上有明确业务流程，工程上有前后端结构、云数据库、云对象和 Git 提交记录。最后冲刺阶段的重点应放在：

1. 补齐提交材料。
2. 完成 H5 部署。
3. 录制演示视频。
4. 做一次完整验收。
5. 避免临近提交前进行高风险大改。
